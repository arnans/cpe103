# Basic Computer Engineering — Course Learning Outcomes (CLOs) & Rubrics

## How these outcomes were designed

Each outcome follows established good practice for learning-outcome design:

- **One verb, one observable behavior.** Each CLO states a single, demonstrable action so it can be taught and assessed cleanly. (The originals often bundled two ideas, e.g. "show skills *and* show understanding.")
- **Measurable action verbs (Bloom's taxonomy).** Vague verbs like *show*, *understand*, or *know* are replaced with verbs that name observable performance — *write, test, apply, analyze, collaborate, communicate, design, integrate*.
- **Condition + criterion where useful.** Outcomes name the context ("using fundamental constructs," "for a defined problem") so the standard of performance is clear.
- **Constructive alignment.** Each CLO maps to how it will be taught and assessed, and the rubric levels make "good" concrete and gradeable.
- **Appropriate cognitive level.** As an introductory ("Basic") course, outcomes target *Apply / Analyze* with a capstone reaching *Create* — a sensible progression.
- **A consistent 4-level performance scale** is used across every rubric:

| Level | Label | Meaning |
|-------|-------|---------|
| 1 | **Beginning** | Below expectations; needs substantial support |
| 2 | **Developing** | Approaching expectations; works with guidance |
| 3 | **Proficient** | Meets expectations; the target standard for the course |
| 4 | **Exemplary** | Exceeds expectations; independent and polished |

> **Level 3 (Proficient) is the pass/target standard.** Level 4 describes work that goes beyond the course requirement.

## The Course Learning Outcomes

> *On successful completion of Basic Computer Engineering, students will be able to:*

| # | Original draft | Revised CLO |
|---|----------------|-------------|
| **CLO1** | Show basic programming skills and show understanding of related important concepts. | **Write, test, and debug basic programs** using fundamental constructs (variables, data types, control flow, functions, and basic data structures), and **explain** the core computing concepts behind them. |
| **CLO2** | Demonstrate how User Journey Analysis and Process Mapping help improve the system design. | **Apply** User Journey Analysis and Process Mapping to a system, and **recommend** design improvements justified by that analysis. |
| **CLO3** | Pursue assignments and projects effectively as a team. | **Collaborate** effectively in a team to plan, execute, and deliver assignments and projects, fulfilling individual responsibilities and contributing to shared goals. |
| **CLO4** | Communicate Engineering concepts both verbally and textually. | **Communicate** engineering concepts clearly and accurately in **written** form and in **oral** presentation, adapting to the audience. |
| **CLO5** | Create a capstone project that demonstrates the ability to apply all principles. | **Design and build** a capstone project that **integrates** programming, system analysis, teamwork, and communication to solve a defined real-world problem. |

Each CLO is detailed below with its rationale, suggested assessment, and a scoring rubric.

## CLO1 — Programming Fundamentals

**Outcome:** Write, test, and debug basic programs using fundamental constructs (variables, data types, control flow, functions, and basic data structures), and explain the core computing concepts behind them.

**Why revised:** "Show … skills" and "show understanding" are not observable. The revision splits the *doing* (write/test/debug) from the *explaining*, and names the concrete constructs so the scope is unambiguous.

**Suggested assessment:** Lab exercises, coding quizzes, a small programming assignment, plus a short oral or written "explain your code" component.

| Criterion | 1 — Beginning | 2 — Developing | 3 — Proficient | 4 — Exemplary |
|-----------|---------------|----------------|----------------|----------------|
| **Correctness** | Program does not run or fails most cases | Runs but fails several cases | Produces correct output for expected cases | Correct, including edge cases |
| **Use of constructs** | Misuses or avoids core constructs | Uses constructs with errors/redundancy | Uses appropriate constructs correctly | Selects efficient, idiomatic constructs |
| **Testing & debugging** | Does not test; cannot locate errors | Tests informally; fixes with help | Tests systematically and fixes own bugs | Anticipates failures; tests proactively |
| **Conceptual explanation** | Cannot explain how the code works | Explains partially or with misconceptions | Accurately explains logic and concepts | Explains, justifies choices, and relates to broader concepts |

## CLO2 — System Analysis (User Journeys & Process Mapping)

**Outcome:** Apply User Journey Analysis and Process Mapping to a system, and recommend design improvements justified by that analysis.

**Why revised:** "Demonstrate how X helps" is hard to grade. The revision makes students *perform* the techniques and *produce a justified recommendation* — the real skill.

**Suggested assessment:** A case-study analysis: students map an existing process / user journey, identify pain points, and propose improvements.

| Criterion | 1 — Beginning | 2 — Developing | 3 — Proficient | 4 — Exemplary |
|-----------|---------------|----------------|----------------|----------------|
| **User Journey Analysis** | Missing or inaccurate | Captures some steps; gaps in stages/touchpoints | Complete, accurate journey with key touchpoints & pain points | Nuanced journey incl. emotions, alternates, edge users |
| **Process Mapping** | Absent or incorrect notation | Partial map; unclear flow | Clear, correct map of the process flow | Precise map with decision points, exceptions, owners |
| **Identifying issues** | No issues identified | Lists issues without evidence | Identifies issues grounded in the analysis | Prioritizes issues by impact with evidence |
| **Design recommendations** | None or irrelevant | Generic suggestions | Specific improvements linked to findings | Improvements with trade-offs and expected impact |

## CLO3 — Teamwork & Collaboration

**Outcome:** Collaborate effectively in a team to plan, execute, and deliver assignments and projects, fulfilling individual responsibilities and contributing to shared goals.

**Why revised:** "Pursue … effectively" is vague. The revision names the collaboration behaviors (plan, execute, deliver, fulfill responsibilities) that can be observed and peer-assessed.

**Suggested assessment:** Peer evaluation, instructor observation, team logs / contribution records, and team deliverables.

| Criterion | 1 — Beginning | 2 — Developing | 3 — Proficient | 4 — Exemplary |
|-----------|---------------|----------------|----------------|----------------|
| **Individual contribution** | Little/no contribution | Inconsistent; needs reminders | Reliably completes assigned share | Goes beyond; supports teammates |
| **Planning & coordination** | Does not engage in planning | Follows plans set by others | Helps plan and track tasks | Drives planning; coordinates effectively |
| **Communication & cooperation** | Disengaged or disruptive | Communicates sporadically | Communicates and cooperates respectfully | Facilitates dialogue; builds consensus |
| **Reliability & conflict handling** | Misses commitments; avoids issues | Meets some deadlines; avoids conflict | Meets commitments; helps resolve issues | Dependable; constructively resolves conflict |

## CLO4 — Technical Communication

**Outcome:** Communicate engineering concepts clearly and accurately in written form and in oral presentation, adapting to the audience.

**Why revised:** Keeps the written + oral intent but adds *clarity, accuracy, and audience adaptation* — the criteria that make communication "good."

**Suggested assessment:** A written report/documentation and an oral presentation or demo, each scored on the rubric.

| Criterion | 1 — Beginning | 2 — Developing | 3 — Proficient | 4 — Exemplary |
|-----------|---------------|----------------|----------------|----------------|
| **Technical accuracy** | Frequent errors/misconceptions | Some inaccuracies | Accurate and correct | Accurate with insightful depth |
| **Written clarity** | Disorganized; hard to follow | Some structure; uneven clarity | Clear, organized, well-structured | Polished, concise, compelling |
| **Oral delivery** | Unclear; reads slides; no engagement | Hesitant; limited engagement | Clear, confident, paced delivery | Engaging; handles questions well |
| **Audience adaptation** | Ignores audience needs | Partial awareness of audience | Adjusts language/detail to audience | Skillfully tailors to technical & non-technical audiences |

---

## CLO5 — Capstone Integration (Create)

**Outcome:** Design and build a capstone project that integrates programming, system analysis, teamwork, and communication to solve a defined real-world problem.

**Why revised:** "Demonstrates the ability to apply all principles" is restated as an integrative *Create*-level outcome that explicitly ties back to CLO1–CLO4, making the capstone the synthesis of the course.

**Suggested assessment:** The capstone deliverable — working artifact + documentation + presentation/demo — assessed individually and as a team.

| Criterion | 1 — Beginning | 2 — Developing | 3 — Proficient | 4 — Exemplary |
|-----------|---------------|----------------|----------------|----------------|
| **Problem definition & scope** | Problem unclear; scope unrealistic | Vague problem or scope | Well-defined problem with feasible scope | Compelling problem; well-justified scope |
| **Technical implementation (CLO1)** | Largely non-functional | Partially works | Works and meets core requirements | Robust, well-built, exceeds requirements |
| **Analysis & design (CLO2)** | No evident analysis | Minimal analysis informs design | Design informed by journey/process analysis | Strong analysis drives thoughtful design |
| **Teamwork (CLO3)** | Uneven/uncoordinated effort | Functional but uneven | Coordinated, balanced collaboration | Exemplary, self-managed collaboration |
| **Communication (CLO4)** | Unclear docs & presentation | Basic docs/presentation | Clear documentation and demo | Professional-quality docs and presentation |
| **Integration & reflection** | Components disconnected | Loosely integrated | Coherent, integrated solution | Seamless integration with reflective insight |

### Quick tips for rollout
- Share the rubrics with students **before** the work begins — rubrics double as instruction.
- Keep the 4-level scale identical across CLOs so grading is consistent and grades are comparable.
- Collect a small amount of evidence per CLO each term; use it to refine the rubric language where graders disagree.
